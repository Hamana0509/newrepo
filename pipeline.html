2Hai phương pháp thực thi

SQL Generation chọn method_used dựa trên metric_type và cấu trúc dữ liệu probe.

Method 1 direct_sql

  • Cho aggregation, distribution, time_series
  • SQL chạy trực tiếp trên BigQuery
  • Kết quả → giải thích ngay
  • Cột phẳng, không cần phân tích phức tạp

Method 2 sql_then_python

  • Cho funnel, cohort, JSON lồng mảng
  • SQL trích xuất → pandas DataFrame
  • Python sandbox: namespace hạn chế, timeout 30s, ≤ 50000 dòng
  • Phân tích cross-session phức tạp

3Schema Probing

Probe SQL rẻ (lọc partition created_at, giới hạn ngày & LIMIT) để biết dữ liệu thật trước khi sinh SQL chính. Auto-chạy nếu hợp lệ + rẻ; ngược lại hiện probe_preview chờ duyệt.

distinct_values

Giá trị thực của cột (service, status, name) + tần suất

temporal_range

Khoảng thời gian + tổng số dòng

null_rate

Tỷ lệ NULL của một cột

json_sample

Mẫu cấu trúc JSON (distribution/funnel/cohort)

sample_rows

Mẫu dòng (funnel/cohort) — PII đã scrub

Lưu ý: tối đa 5 probe/run (SUBAGENT_MAX_PROBES), cửa sổ 7 ngày (SUBAGENT_PROBE_DAYS). Probe lỗi không raise — degrade gracefully. probe_text cap 4000 ký tự để tránh MALFORMED_FUNCTION_CALL.

4State Machine

Quản lý ở models/conversation.py — in-memory, key theo conversation_id (mất khi restart).

IDLE→ PLANNING→ AWAITING_PROBE_CONFIRM→ AWAITING_CONFIRM→ EXECUTING→ IDLE
AWAITING_PROBE_CONFIRM chỉ xuất hiện khi probe đắt hoặc không hợp lệ — phần lớn trường hợp probe auto-chạy và bỏ qua state này.

6Guardrails & Tracing

🛡️ SQL Validation

  • Chỉ cho SELECT — chặn mọi DML
  • Whitelist 7 bảng trong bef-cake-prod
  • Chặn 9 cột PII (phone, full_name, document_id…)
  • Gotcha: tb_audit join qua ref_id

📊 Pipeline Tracing

  • StepTrace → PipelineTrace mỗi run
  • Commit sau khi user accept/reject (hoặc lỗi)
  • Xem: GET /api/conversations/{id}/traces
  • LLM log thô: GET /api/llm-logs?last=N

⚠️ Error Handling

  • Map sang thông điệp tiếng Việt thân thiện
  • classify_error(): 503 quá tải, 429 rate limit, hết phiên GCloud
  • Ghi pipeline_error → commit → về IDLE
